## 降级策略与容错链路

本文档描述 StockVision 平台在各级服务不可用时的降级路径与容错机制。系统设计了三层降级防线，确保用户在部分甚至全部后端服务离线时仍能看到图表和分析结果。

---

### 整体降级架构

系统从上到下有 **三层降级**，每一层独立运作、逐级兜底：

```
┌──────────────────────────────────────────────────────────────┐
│  第一层：后端数据源主备切换（data_provider.py）                │
│  腾讯证券 ⇄ Yahoo Finance — 对上层完全透明                    │
├──────────────────────────────────────────────────────────────┤
│  第二层：前端 API 降级（api.ts）                              │
│  后端 K 线接口失败 → 前端直连腾讯；后端分析接口失败 → mockEngine│
├──────────────────────────────────────────────────────────────┤
│  第三层：演示模式（App.vue + api.ts，后端离线自动进入）                         │
│  后端不可达 → 前端随机游走生成模拟 K 线 + mockEngine 分析    │
└──────────────────────────────────────────────────────────────┘
```

---

### 第一层：后端数据源主备切换

**所在文件：** `backend/data_provider.py`

**核心类：** `DataProviderManager`

**切换策略：**

| K 线周期 | 主源 | 备源 | 选择理由 |
|----------|------|------|----------|
| `4h` | Yahoo Finance | 腾讯证券 | Yahoo 的 1h 历史数据更丰富（每天约 7 根，保留 730 天），合成 4h 数据量更充足 |
| `1d` | 腾讯证券 | Yahoo Finance | 腾讯日K响应快、无需额外依赖 |
| `1w` | 腾讯证券 | Yahoo Finance | 同上 |
| `1M` | 腾讯证券 | Yahoo Finance | 同上 |
| `3M`/`6M`/`1Y` | 腾讯日线 | Yahoo 1d | 季/半年/年K 由日线聚合，两源均拉日线后在后端重采样 |

**故障转移流程：**

```
DataProviderManager.fetch_klines(symbol, period)
  │
  ├─ 确定主/备源顺序
  │
  ├─ 调用 primary.fetch_klines()
  │    └─ 返回非空 DataFrame → 记录 last_source, 返回数据 ✓
  │
  ├─ 主源失败（空 DataFrame）→ 调用 backup.fetch_klines()
  │    └─ 返回非空 DataFrame → 记录 last_source, 返回数据 ✓
  │
  └─ 全部失败 → 抛出 HTTPException(404)
```

**各数据源的失败判定：**

- **TencentProvider：** HTTP 请求异常（httpx 超时/网络错误）、响应 code ≠ 0、解析后数据为空、有效行数 < 5 —— 均返回空 DataFrame，不抛异常
- **YahooProvider：** yfinance 未安装（ImportError）、Ticker 转换失败、history 返回空 DataFrame、过滤无效行后 < 5 行 —— 均返回空 DataFrame，不抛异常

**关键设计决策：**

1. 单个数据源失败 **不抛异常**，只返回空 DataFrame，由 Manager 决定是否切换
2. `last_source` 属性记录最近成功的数据源名称，供后端响应 `data_source` 字段和调试日志使用
3. YahooProvider 使用 `asyncio.to_thread()` 将同步的 yfinance 调用放入线程池，避免阻塞 FastAPI 的事件循环
4. 4h 重采样逻辑 `_resample_to_4h()` 被两个数据源共享（YahooProvider 直接调用 TencentProvider 的静态方法）

#### 重采样前日线粒度校验守卫（季/半年/年K 专用）

季K(3M)/半年K(6M)/年K 需要把**日线**聚合为大周期，但某些环境下数据源会退化——例如 Yahoo 代理偶尔把 `interval=1d` 退化为"每年一根"的聚合桩数据。若直接拿退化数据聚合，年K 会出现拆股/跳跃缺口。

**守卫逻辑（在 `_dispatch_resample` / TencentProvider 重采样入口）：**

```
validate_klines(df, '1d')  # 用相邻日期中位数间隔判断：日线 ≤5 天
  ├─ 通过 (确为日线) → apply_forward_adjustment(df) → _resample_to_period(df, rule)
  └─ 不通过 (退化/聚合桩) → 返回空 DataFrame，Manager 拒绝该源并回退到另一源
```

`validate_klines(df, period)` 的粒度阈值（相邻日期中位数间隔上限）：
`1d≤5天 / 1w≤14天 / 1M≤45天 / 3M≤100天 / 6M≤200天 / 1Y≤400天`。
本地 SQLite 缓存读取后也会再做一次 `validate_klines`，污染缓存视为未命中、触发重新拉取。

**设计决策：**
- 这是"数据正确性的最后一道防线"：宁可该周期返回 404，也不要把错标/缺口数据展示给用户
- Yahoo 直连被守卫拒绝后，Manager 自动回退到 Yahoo `yfinance` 库（返回真实日线）或腾讯日线，最终仍能聚出正确大周期蜡烛
- 沙箱/代理环境下若所有源都退化，3M/6M/1Y 会统一回退到腾讯日线（真实粒度但仅约 8 年，非全量历史）

### 第二层：前端 API 降级

**所在文件：** `frontend/src/utils/api.ts`

#### 2a. K 线数据降级（getKLines 函数）

```
getKLines(symbol, period)
  │
  ├─ fetch('/api/klines') → 成功 → 返回数据 ✓
  │
  └─ 失败（后端不可达/返回错误）
       │
       ├─ period === '4h' → 直接抛错（4h 无法前端直连，需后端合成）
       │    错误信息: "4小时K线需后端支持，请确认后端服务已启动"
       │
       └─ period ∈ {'1d','1w','1M'} → 尝试前端直连腾讯
            │
            ├─ fetch('https://web.ifzq.gtimg.cn/...') → 成功解析 → 返回数据 ✓
            │
            └─ 腾讯直连也失败 → 抛错
                 错误信息: "网络原因，无法加载 K 线行情图"
```

**前端直连腾讯的技术细节：**

- 仅对非 4h 周期生效（4h 需要后端从 1h 数据重采样，前端无法直接获取）
- 直连 URL 与后端不同：使用 `web.ifzq.gtimg.cn/appzone/newstrend/new_chart/query` 端点（腾讯新版趋势接口），而后端使用 `web.ifzq.gtimg.cn/appstock/app/fqkline/get` 端点
- 股票代码需转换为小写（如 `usaapl`），不带交易所后缀
- 返回数据格式为 `data.{ticker}.qfqk`，字段顺序为 `[日期, 开, 收, 高, 低, 成交量]`（注意：与后端腾讯源的字段顺序不同，后端是 `[日期, 开, 收, 高, 低, 量]`，但前端直连的解析代码中 open=item[1], close=item[2], high=item[3], low=item[4]）
- 数据量要求 ≥ 5 条，否则视为数据为空

**重要差异：** 前端直连腾讯使用的是与后端不同的 API 端点，这意味着两者的数据格式和字段含义可能存在细微差异。如果腾讯调整了某一端点的返回格式，另一端不一定同步变化。

#### 2b. 分析计算降级（getStockAnalysis 函数）

```
getStockAnalysis(symbol, startDate, endDate, klineCache, period)
  │
  ├─ fetch('/api/analysis', POST) → 成功 → 返回分析结果 ✓
  │
  └─ 失败（后端不可达/计算报错）
       │
       └─ 降级到前端 mockEngine 计算
            │
            ├─ 优先使用 klineCache 中已有的 K 线数据（避免重复请求）
            │
            ├─ klineCache 为空 → 调用 getKLines() 获取（此调用本身也有降级）
            │
            ├─ K 线数据为空 → 抛错 "无法提取 K 线行情进行分析"
            │
            └─ 调用 mockEngine.calculateSRLevels() + mockEngine.calculateRangeStats()
                 → 返回与后端相同结构的 AnalysisResponse ✓
```

**关键设计决策：**

1. 前端 mockEngine 是后端 `analysis.py` 的 **1:1 TypeScript 移植**，算法逻辑完全一致（详见「核心算法规格说明.md」中的"前后端一致性"章节）
2. 降级计算时优先复用已缓存的 K 线数据，避免在网络不稳定的情况下再次请求失败
3. 降级返回的数据结构与后端完全一致（`sr_levels` + `statistics`），上层 App.vue 无需区分数据来源

### 第三层：演示模式（完全离线，自动判定）

**所在文件：** `frontend/src/App.vue`（`useMockData` 状态）+ `frontend/src/utils/api.ts`（`generateMockKLines` 函数）

**触发方式：** 由 `detectBackend()` **自动判定，无手动开关**。前端挂载时请求 `GET /api/health`：
- 后端可达（200）→ `useMockData = false`（实时模式）
- 后端不可达 / 2 秒超时 → `useMockData = true`（演示模式，随机游走模拟数据）

> 注：2026-07-15 起已移除原「测试模式」手动切换开关。单独启动前端（无后端）即自动进入演示模式；启动后端后刷新页面或点底部「后端已启动？重试连接」即自动切回实时。底部「重试连接」仅重新探测 + 刷新，并非模式切换按钮。

**运作流程：**

```
handleSearch() 被调用
  │
  ├─ useMockData === true
  │    │
  │    ├─ 根据股票代码查 basePriceMap 获取基准价格
  │    │   AAPL=180, NVDA=450, TSLA=250, MSFT=370, GOOGL=140,
  │    │   AMZN=175, META=480, AMD=160, MU=130, SNDK=95, 其他=150
  │    │
  │    ├─ 调用 generateMockKLines(code, 4500, basePrice)
  │    │   → 随机游走算法生成 ~4500 天（≈18 年）的模拟日K
  │    │
  │    ├─ 调用 resampleMockKLines(klines, currentPeriod)
  │    │   → 按所选周期（日/周/月/季/半年/年）聚合为对应粒度 K 线
  │    │
  │    └─ 调用 runAnalysis()
  │         └─ useMockData === true → 使用 mockEngine 本地计算
  │              → calculateSRLevels() + calculateRangeStats()
  │
  └─ useMockData === false → 正常 API 调用流程
```

**随机游走算法（generateMockKLines）：**

- 每日价格变动：`change = (Math.random() - 0.5) * 2 * volatility`
- 开盘价 = 前一日收盘价
- 收盘价 = 开盘价 × (1 + change)
- 最高价 = max(开, 收) × (1 + Math.random() × volatility × 0.5)
- 最低价 = min(开, 收) × (1 - Math.random() × volatility × 0.5)
- 成交量 = 500 万 ~ 5500 万之间的随机整数
- 自动跳过周末（周六/周日）
- 每只股票有独立的波动率参数（volatilityMap）

**测试模式的限制：**

- 生成**日线**后经 `resampleMockKLines` 支持全部展示周期（日K/周K/月K/季K/半年K/年K），但**不支持 4h**（需 1h 数据合成，测试模式只产日线）
- 数据完全是随机的，不包含真实市场特征（如趋势、季节性等）
- 搜索功能仍依赖后端（`searchStocks` 没有降级逻辑），如果后端离线则无法搜索新股票

### 缓存机制与降级的交互

前端有两层缓存，在降级场景中起重要作用：

| 缓存 | 键格式 | 存储内容 | 降级时的作用 |
|------|--------|----------|-------------|
| `klineCache` | `{symbol}_{period}` | K 线数据数组 | 分析降级时优先复用缓存的 K 线数据，避免再次网络请求 |
| `analysisCache` | `{symbol}_{period}_{startDate}_{endDate}` | 分析结果（SR + 统计） | 切换周期/重新搜索时，如果缓存命中直接复用，不触发任何网络请求。**键含起止日期**（v0.3.0）：区间统计结果依赖日期区间，改日期后必须 cache miss 重新计算，否则面板不更新 |

**缓存命中时的行为：** 如果 `klineCache` 或 `analysisCache` 中已有对应数据，系统直接使用缓存，完全不发起网络请求。这意味着即使后端在首次加载后宕机，用户切换回已查看过的股票仍可看到完整数据和分析。

### 降级链路全景图

```
用户搜索股票 handleSearch()
  │
  ├─[演示模式/后端离线] → generateMockKLines() → mockEngine 分析 → 显示结果
  │
  └─[正常模式]
       │
       ├─ 检查 klineCache → 命中 → 跳过网络请求
       │
       └─ 未命中 → getKLines()
            │
            ├─ /api/klines → 后端
            │    │
            │    ├─ 后端 DataProviderManager
            │    │    ├─ 主源成功 → 返回 ✓
            │    │    └─ 主源失败 → 备源成功 → 返回 ✓
            │    │    └─ 全失败 → HTTP 404
            │    │
            │    └─ 后端返回数据 ✓
            │
            └─ /api/klines 失败（后端离线）
                 │
                 ├─ 4h → 报错（无法降级）
                 │
                 └─ 1d/1w/1M → 前端直连腾讯
                      ├─ 成功 → 返回 ✓
                      └─ 失败 → 报错
       
       K 线获取后 → runAnalysis()
         │
         ├─ 检查 analysisCache → 命中 → 跳过
         │
         └─ 未命中 → getStockAnalysis()
              │
              ├─ /api/analysis → 后端计算 → 返回 ✓
              │
              └─ /api/analysis 失败 → mockEngine 本地计算
                   ├─ 使用 klineCache 或重新获取 K 线
                   └─ 返回计算结果 ✓
```

### 各降级场景下的用户体验

| 场景 | K 线数据 | 分析结果 | 用户感知 |
|------|----------|----------|----------|
| 全部正常 | 后端（腾讯/Yahoo） | 后端计算 | 无任何异常 |
| 腾讯不可用 | 后端自动切 Yahoo | 后端计算 | 无感知（data_source 字段变为 "yahoo"） |
| Yahoo 不可用 | 后端自动用腾讯 | 后端计算 | 无感知 |
| 两个数据源都不可用 | HTTP 404 | — | 错误提示 |
| 后端完全离线（日/周/月线） | 前端直连腾讯 | mockEngine 本地计算 | 图表正常显示，分析结果正常（来源不同但算法一致） |
| 后端完全离线（4h 线） | 失败 | — | 错误提示："4小时K线需后端支持" |
| 演示模式（后端离线） | 随机游走模拟数据 | mockEngine 本地计算 | UI 标注"演示模式"，数据非真实行情 |
| 后端先正常后离线 | 缓存命中 | 缓存命中 | 完全无感知，已查看过的股票不受影响 |

### 搜索接口的降级缺失

当前 `searchStocks()` 函数 **没有降级逻辑**：

```typescript
// api.ts 第 61-71 行
export async function searchStocks(keyword: string): Promise<StockInfo[]> {
  try {
    const response = await fetch(url);
    if (!response.ok) throw new Error(...);
    return await response.json();
  } catch (error) {
    throw new Error('网络连接异常，搜不到该股票');
  }
}
```

如果后端离线，搜索功能完全不可用。用户只能通过已加载的股票继续操作，无法搜索新的股票。这是一个潜在的改进点——可以在前端内置一个本地股票列表作为降级。

### 修改降级策略时的检查清单

- 修改 `DataProviderManager` 的主备策略 → 检查 `last_source` 是否正确更新，检查 `data_source` 字段
- 修改前端直连腾讯的 URL 或解析逻辑 → 对比后端 `TencentProvider._fetch_raw()` 的数据格式，确保字段映射一致
- 修改 `mockEngine` 的算法 → 同步修改后端 `analysis.py`（或反过来），保持 1:1 一致
- 修改 `generateMockKLines` 的波动率参数 → 检查 App.vue 中的 `basePriceMap` 是否需要同步更新
- 添加新的 K 线周期 → 需要在 6 个位置同步更新：`PERIOD_MAP`（后端 data_provider.py）、`SUPPORTED_PERIODS`（后端 + 前端 api.ts）、`PERIOD_LABELS`（前端 api.ts）、`PERIOD_PARAMS`（后端 analysis.py + 前端 mockEngine.ts）；若是聚合周期（季/半年/年）还需在 `PERIOD_MAP` 设 `resample` 频率码（`QS`/`2QS`/`YS`）并在 `validate_klines` 增加对应粒度阈值
- 修改重采样频率码 → 必须用起始日锚定的 `QS`/`2QS`/`YS`，严禁 `Q`/`2Q`/`YE`（见「技术债与已知陷阱.md」第 14 条）
- 修改缓存策略 → 检查 `klineCache` 和 `analysisCache` 的键格式是否一致（`analysisCache` 键含起止日期，见「缓存机制」表）
- 修改测试模式生成或周期逻辑 → 同步 `mockEngine.resampleMockKLines`，确保覆盖所选周期（否则周K/月K 等显示成日K，见「技术债与已知陷阱.md」第 16 条）
- 修改 `data_provider.py` / `analysis.py` / `main.py` → **必须重启后端进程** 才生效（HMR 只适用于前端，见「技术债与已知陷阱.md」第 17 条）
