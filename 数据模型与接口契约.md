# 数据模型与接口契约

> 本文档集中记录 StockVision 前后端所有数据结构定义、API 接口契约（请求/响应格式、状态码、边界条件），帮助 AI 在修改接口时保持前后端一致。

---

## 1. 后端 Pydantic 数据模型

定义位置：`stage_one/backend/main.py`

### 1.1 KLinePoint — K线数据点

```python
class KLinePoint(BaseModel):
    time: str          # 日/周/月线: 'YYYY-MM-DD' | 4小时线: 'YYYY-MM-DD HH:MM'
    open: float        # 开盘价
    high: float        # 最高价
    low: float         # 最低价
    close: float       # 收盘价
    volume: float      # 成交量
```

**约束与约定：**
- `time` 字段格式由 K 线周期决定：日/周/月线为纯日期字符串，4h 线包含时分
- 所有价格字段为浮点数，无精度约束（由数据源原始精度决定）
- `volume` 为浮点数（非整数），因数据源可能返回浮点成交量

### 1.2 AnalysisRequest — 分析请求

```python
class AnalysisRequest(BaseModel):
    symbol: str          # 股票代码，如 'usAAPL'
    start_date: str      # 区间开始，格式 'YYYY-MM-DD' 或 'YYYY-MM-DD HH:MM'
    end_date: str        # 区间结束，格式同上
    period: str = "1d"   # K线周期: '4h' / '1d' / '1w' / '1M' / '3M' / '6M' / '1Y'
```

**约束与约定：**
- `symbol` 必须带 `us` 前缀（如 `usAAPL`），后端不会自动添加
- `period` 默认 `1d`，若传入不支持的周期会被静默替换为 `1d`（main.py L191）
- `start_date` / `end_date` 用于区间统计，不影响支撑阻力位计算（SR 使用全部 K 线数据）

### 1.3 SRLevel — 支撑/阻力价位

```python
class SRLevel(BaseModel):
    price: float    # 价位
    count: int      # 该价位聚类中的极值点数量
    type: str       # 'support' 或 'resistance'
    stars: int      # 强度星级: 2(Weak) / 3(Medium) / 5(Strong)
    level: str      # 'Weak' / 'Medium' / 'Strong'
```

**约束与约定：**
- `stars` 只取 2、3、5 三个值（不是连续的）
- `type` 由最新价格决定：价位 < 最新价 → support，价位 > 最新价 → resistance
- 返回顺序：前 3 个为阻力位（按 score 降序），后 3 个为支撑位（按 score 降序）
- 内部计算时还有 `weight` 和 `score` 字段，但不会出现在 API 响应中

### 1.4 RangeStat — 区间统计单项

```python
class RangeStat(BaseModel):
    pct: float       # 涨跌幅百分比 (如 +32.61 表示涨 32.61%)
    start: str       # 起点日期
    end: str         # 终点日期
```

### 1.5 AnalysisResponse — 分析响应

```python
class AnalysisResponse(BaseModel):
    sr_levels: List[SRLevel]           # 支撑阻力位列表 (最多6个: 3阻力+3支撑)
    statistics: Dict[str, RangeStat]   # {"max_rise": RangeStat, "max_fall": RangeStat}
    period: str = "1d"                 # 实际使用的K线周期
    data_source: str = ""              # 实际命中的数据源名称 ("tencent" / "yahoo" / "")
```

**约束与约定：**
- `sr_levels` 可能为空列表（数据不足时）
- `statistics` 始终包含 `max_rise` 和 `max_fall` 两个 key
- `data_source` 为空字符串表示两个数据源都未成功（此时会抛 404）

### 1.6 StockInfo — 股票信息（字典，非 Pydantic）

```python
# 以字典列表形式存在于 STOCKS_POOL 和 API 响应中
{
    "symbol": "usAAPL",      # 带 us 前缀的代码
    "name": "苹果 Apple",     # 中英文名称
    "pinyin": "pg"           # 拼音缩写 (仅股票池中有)
}
```

## 2. 前端 TypeScript 接口

### 2.1 api.ts 中的接口定义

```typescript
// 股票信息
interface StockInfo {
  symbol: string;
  name: string;
  pinyin?: string;        // 可选，搜索建议中不总是返回
}

// K线数据点 — 与后端 KLinePoint 完全对应
interface KLinePoint {
  time: string;
  open: number;
  high: number;
  low: number;
  close: number;
  volume: number;
}

// 支撑/阻力位 — 与后端 SRLevel 完全对应
interface SRLevel {
  price: number;
  count: number;
  type: 'support' | 'resistance';   // 联合类型，比后端更严格
  stars: number;
  level: 'Weak' | 'Medium' | 'Strong';  // 联合类型
}

// 区间统计单项
interface RangeStatPoint {
  pct: number;
  start: string;
  end: string;
}

// 区间统计
interface RangeStats {
  max_rise: RangeStatPoint;
  max_fall: RangeStatPoint;
}

// 分析响应 — 注意：比后端 AnalysisResponse 少了 period 和 data_source
interface AnalysisResponse {
  sr_levels: SRLevel[];
  statistics: RangeStats;
}
```

### 2.2 mockEngine.ts 中的镜像接口

```typescript
// mockEngine.ts 独立定义了以下接口 (与 api.ts 字段完全相同):
interface KLinePoint { ... }      // 同 api.ts
interface SRLevel { ... }         // 同 api.ts
interface RangeStatPoint { ... }  // 同 api.ts
interface RangeStats { ... }      // 同 api.ts
```

**重要：** `KLinePoint` 和 `SRLevel` 在 `api.ts` 和 `mockEngine.ts` 中各有一份独立定义。修改字段时必须两处同步，否则 TypeScript 类型检查会报错或运行时数据不匹配。

## 3. 前后端模型对照表

| 后端 (Python) | 前端 (TypeScript) | 字段差异 |
|---|---|---|
| `KLinePoint.time: str` | `KLinePoint.time: string` | 无 |
| `KLinePoint.open: float` | `KLinePoint.open: number` | Python float ↔ TS number |
| `SRLevel.type: str` | `SRLevel.type: 'support' \| 'resistance'` | 前端用联合类型约束 |
| `SRLevel.level: str` | `SRLevel.level: 'Weak' \| 'Medium' \| 'Strong'` | 前端用联合类型约束 |
| `RangeStat` | `RangeStatPoint` | **命名不同** |
| `AnalysisResponse.period: str` | (前端接口无此字段) | **前端缺少此字段** |
| `AnalysisResponse.data_source: str` | (前端接口无此字段) | **前端缺少此字段** |
| `AnalysisRequest.symbol: str` | (JSON body 内联构造) | 前端无独立类型定义 |

## 4. REST API 接口契约

### 4.1 GET /api/search

**用途：** 模糊搜索股票

**请求参数：**
| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `keyword` | string | 否 | 搜索关键词（股票名称/代码/拼音缩写） |

**响应：** `StockInfo[]` (JSON 数组)

**行为逻辑：**
1. 无 keyword → 返回完整股票池 (10 只)
2. 有 keyword → 在股票池中模糊匹配 (symbol/name/pinyin 包含匹配)
3. 池中无匹配 + keyword 以 `us` 开头 → 返回 `[{"symbol": "usXXX", "name": "美股 XXX"}]`
4. 池中无匹配 + keyword 为纯字母 → 返回 `[{"symbol": "usXXX", "name": "美股 XXX"}]`
5. 以上都不满足 → 返回空数组 `[]`

**状态码：** 200 (始终返回 200，即使无匹配)

---

### 4.2 GET /api/klines

**用途：** 获取多周期 K 线数据

**请求参数：**
| 参数 | 类型 | 必填 | 默认值 | 说明 |
|---|---|---|---|---|
| `symbol` | string | 是 | - | 股票代码，如 `usAAPL` |
| `period` | string | 否 | `1d` | K 线周期: `1d` / `1w` / `1M` / `3M` / `6M` / `1Y`（`4h` 后端仍支持但前端隐藏） |

**响应：** `KLinePoint[]` (JSON 数组，**全量历史**，根数取决于周期：日K 数千根、月K 数百根、年K 数十根)

**状态码：**
| 状态码 | 条件 |
|---|---|
| 200 | 成功 |
| 400 | period 不在 SUPPORTED_PERIODS 中 |
| 404 | 所有数据源均无法获取数据（含重采样前日线粒度校验失败） |

**数据源路由：**
- `3M` / `6M` / `1Y`（季K/半年K/年K）→ 拉取**日线**后由后端按日历时间聚合：`Tencent day` 优先 → `Yahoo 1d` 备选；重采样前先校验原始确为日线粒度，否则拒绝并回退
- `1d` / `1w` / `1M` → Tencent 优先 → Yahoo 备选
- `4h` → Yahoo 优先 → Tencent 备选（仅后端，前端 UI 已隐藏该周期）

---

### 4.3 POST /api/analysis

**用途：** 获取支撑阻力位分析 + 区间涨跌统计

**请求体 (JSON)：**
```json
{
  "symbol": "usAAPL",
  "start_date": "2024-01-01",
  "end_date": "2024-12-31",
  "period": "1d"
}
```

**响应：** `AnalysisResponse` (JSON 对象)

**状态码：**
| 状态码 | 条件 |
|---|---|
| 200 | 成功 |
| 404 | 获取 K 线数据失败 |

**内部流程：**
1. 获取 K 线 DataFrame（全量历史，按 period 拉取/重采样）
2. `calculate_sr_levels(df, period)` → 全量 K 线计算支撑阻力
3. `calculate_range_stats(df, start_date, end_date)` → 指定区间统计涨跌幅
4. 返回组合结果

---

### 4.4 GET /api/periods

**用途：** 获取支持的 K 线周期列表

**请求参数：** 无

**响应：**（返回的是**后端** `SUPPORTED_PERIODS`，含 `4h`；前端 UI 用自身常量 `['1d','1w','1M','3M','6M','1Y']` 渲染周期按钮，已隐藏 `4h`）
```json
{
  "periods": ["4h", "1d", "1w", "1M", "3M", "6M", "1Y"],
  "descriptions": {
    "4h": "4小时K线 (短期波段, 仅后端)",
    "1d": "日K线 (默认)",
    "1w": "周K线 (中期趋势)",
    "1M": "月K线 (长期结构)",
    "3M": "季K线 (每根=一个季度)",
    "6M": "半年K线 (每根=半年)",
    "1Y": "年K线 (每根=一年)"
  }
}
```

**状态码：** 200 (始终)

## 5. 股票代码格式约定

系统中存在多种代码格式，转换规则如下：

```
用户输入 (App.vue)     →   API 请求格式        →   腾讯代码              →   Yahoo ticker
─────────────────────────────────────────────────────────────────────────────────────────
AAPL (纯大写无前缀)     →   usAAPL               →   usAAPL.OQ/.N/.AM     →   AAPL
usAAPL (带前缀)         →   usAAPL               →   usAAPL.OQ/.N/.AM     →   AAPL
usAAPL.OQ (带后缀)      →   usAAPL.OQ            →   usAAPL.OQ (直接用)    →   AAPL
```

**前端代码格式：**
- `symbolInput` / `currentSymbol`: 纯大写无前缀 (如 `AAPL`)
- API 调用时自动拼接 `us` 前缀: `'us' + code`
- 自选股列表存储纯大写代码 (如 `AAPL`, `TSLA`)

**后端代码格式：**
- API 接收带 `us` 前缀的代码 (如 `usAAPL`)
- 腾讯数据源自动尝试三个交易所后缀: `.OQ` (纳斯达克), `.N` (纽交所), `.AM` (美交所)

## 6. 时间格式约定

| 场景 | 格式 | 示例 |
|---|---|---|
| 日/周/月线 `time` 字段 | `YYYY-MM-DD` | `2024-03-15` |
| 4h 线 `time` 字段 | `YYYY-MM-DD HH:MM` | `2024-03-15 10:00` |
| 季K(3M) `time` 字段 | `YYYY-MM-DD`（季度**起点**） | `2024-01-01` / `2024-04-01` / `2024-07-01` / `2024-10-01` |
| 半年K(6M) `time` 字段 | `YYYY-MM-DD`（半年**起点**） | `2024-01-01` / `2024-07-01` |
| 年K(1Y) `time` 字段 | `YYYY-MM-DD`（年**起点**） | `2024-01-01` |
| 前端图表内部 | UNIX 时间戳 (秒, number) | `1710489600` |
| 区间统计日期比较 | 字符串直接比较 | `'2024-01-01' >= '2024-01-01'` |

**聚合周期标签规则（重要）：** 季K/半年K/年K 由日线 `df.resample()` 聚合，日期标签统一对齐**周期起点**（使用起始日锚定的频率码 `QS`/`2QS`/`YS`）。例如一根年K蜡烛的 `time` 是该年 1 月 1 日，代表这一整年的 OHLCV 聚合。切勿使用已废弃的 `Q`/`2Q`/`YE` 码——它们会产生季末标签并冒出一个只含首日的"前导空桶"。

**时间转换函数：** `toChartTime(timeStr: string): number` — 将各种格式统一转为 UNIX 秒级时间戳，供 lightweight-charts 使用。这是前端时间处理的核心函数，处理了日线字符串、4h 带时分字符串、ISO 格式等多种输入。

## 7. 缓存键约定

前端使用 `symbol_period` 组合作为缓存键：

```typescript
const cacheKey = (symbol: string, period: Period) => `${symbol}_${period}`;
// 示例: "usAAPL_1d", "usTSLA_4h"
```

- `klineCache`: `Map<string, KLinePoint[]>` — K 线数据缓存
- `analysisCache`: `Map<string, AnalysisResponse>` — 分析结果缓存
- 缓存生命周期：页面会话期间，无过期机制，刷新页面清空
