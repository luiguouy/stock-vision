## 技术债与已知陷阱

本文档记录 StockVision 项目中已知的技术债务、设计妥协和容易踩坑的地方。AI Agent 在修改代码前应仔细阅读，避免重复踩坑或引入回归。

---

### 1. App.vue 巨型单文件组件

**严重程度：** 高
**位置：** `frontend/src/App.vue`（约 1718 行）

整个前端应用的模板、逻辑和样式全部集中在一个 Vue 单文件组件中。这包括搜索逻辑、K 线加载、图表初始化、支撑阻力分析、自选股管理、深色模式、键盘快捷键、时间范围选择等所有功能。

**带来的问题：**

- 修改任何功能都需要在 1700+ 行中定位上下文，AI Agent 容易遗漏相关状态或 watcher
- 无法对单个功能模块进行独立测试
- 多个关注点（UI 渲染、数据获取、算法计算、图表操作）混杂在一起
- `<script setup>` 中声明的变量和函数无法被其他组件复用

**建议拆分方向：** 如果未来重构，可以考虑拆为：`useStockSearch`（搜索逻辑）、`useChartData`（K 线加载与缓存）、`useAnalysis`（分析计算与降级）、`useChart`（图表初始化与交互）、`useWatchlist`（自选股管理）等 Composable 函数。

### 2. TypeScript 接口重复定义

**严重程度：** 中
**位置：** `frontend/src/utils/api.ts` 和 `frontend/src/utils/mockEngine.ts`

以下接口在两个文件中各定义了一次：

| 接口名 | api.ts 行号 | mockEngine.ts 行号 |
|--------|-------------|-------------------|
| `KLinePoint` | 25-32 | 5-12 |
| `SRLevel` | 34-40 | 14-20 |
| `RangeStatPoint` | 42-46 | 22-26 |
| `RangeStats` | 48-51 | 28-31 |

**风险：** 如果修改了其中一个文件的接口定义而忘记同步另一个，TypeScript 编译器不会报错（因为两个文件各自导出同名接口），但运行时的数据交换会出现字段缺失或类型不匹配。

**建议：** 将共享接口提取到 `frontend/src/types.ts`，两个文件统一从该文件导入。

### 3. 前后端数据模型不一致

**严重程度：** 中
**位置：** 后端 `main.py` 的 Pydantic 模型 vs 前端 TypeScript 接口

| 差异点 | 后端 | 前端 | 影响 |
|--------|------|------|------|
| 区间统计模型名 | `RangeStat` | `RangeStatPoint` | 命名不一致，阅读代码时容易混淆 |
| 分析响应额外字段 | `AnalysisResponse` 包含 `period` 和 `data_source` | 前端 `AnalysisResponse` 不含这两个字段 | 后端返回的 `period` 和 `data_source` 在前端被静默忽略，不影响功能但丢失了有用的调试信息 |
| statistics 类型 | `Dict[str, RangeStat]` | `RangeStats`（固定结构） | 后端用 Dict 泛型，前端用具体接口，结构兼容但类型表达不同 |

**风险：** 如果后端新增字段或删除字段，前端不会有任何编译时提示。

### 4. YahooProvider 使用线程池执行同步代码

**严重程度：** 低
**位置：** `backend/data_provider.py` 第 250 行

```python
df = await asyncio.to_thread(
    self._fetch_yfinance_sync, yf, ticker, yahoo_interval, fetch_count, need_resample
)
```

yfinance 是纯同步库，会阻塞事件循环。当前通过 `asyncio.to_thread()` 放入线程池解决。这在低并发场景下工作正常，但如果多个用户同时请求且主源腾讯失败触发 Yahoo 降级，线程池可能成为瓶颈。

**注意事项：**

- `asyncio.to_thread()` 使用默认线程池（`ThreadPoolExecutor`），默认 `min(32, os.cpu_count() + 4)` 个线程
- yfinance 的网络请求没有设置超时参数，如果 Yahoo 服务器无响应，线程会被长时间占用
- 每次调用都会 `import yfinance as yf`（第 235 行），虽然有模块缓存但写法不够优雅

### 5. 前端直连腾讯 API 绕过后端

**严重程度：** 中
**位置：** `frontend/src/utils/api.ts` 第 89-111 行

当后端 K 线接口不可用时，前端会直接请求腾讯证券 API。这带来几个问题：

- **CORS 风险：** 腾讯 API 没有对浏览器跨域请求做特殊处理，如果腾讯将来添加 CORS 限制，前端直连将失效
- **端点不同：** 前端使用 `/appzone/newstrend/new_chart/query`，后端使用 `/appstock/app/fqkline/get`，两个端点的数据格式可能不同步变化
- **字段顺序差异：** 前端直连解析顺序为 `[日期, 开, 收, 高, 低, 量]`（`item[1]=open, item[2]=close`），后端解析顺序也是 `[日期, 开, 收, 高, 低, 量]`，但两个 API 端点的实际返回格式是否一致未经严格验证
- **无数据源切换日志：** 前端直连成功后没有记录 `data_source`，后端响应中的 `data_source` 字段在此场景下无意义

### 6. 股票代码格式转换链路

**严重程度：** 中（容易出错）
**位置：** 多个文件

代码在不同位置使用不同的股票代码格式，转换逻辑分散在多处：

```
用户输入: "AAPL" 或 "aapl"
    ↓ App.vue handleSearch()
前端内部: "AAPL"（currentSymbol，纯大写无前缀）
    ↓ api.ts getKLines/getStockAnalysis
API 请求: "usAAPL"（带 us 前缀）
    ↓ main.py → data_provider.py
腾讯格式: "usAAPL.OQ" / "usAAPL.N" / "usAAPL.AM"（自动尝试三个后缀）
    ↓ YahooProvider._to_yahoo_ticker()
Yahoo格式: "AAPL"（去前缀去后缀）
    ↓ mockEngine
前端降级: "AAPL"（直接使用，不需要转换）
```

**已知陷阱：**

- App.vue 中 `currentSymbol` 存储的是无前缀格式（如 `AAPL`），但调用 API 时需要加 `us` 前缀。这个转换在 `handleSearch()` 和 `runAnalysis()` 中分别执行，逻辑略有不同
- `runAnalysis()` 第 1435 行的转换逻辑：`const querySymbol = currentSymbol.value.startsWith('US') ? currentSymbol.value : 'us' + currentSymbol.value;` — 如果 `currentSymbol` 已经是 `usAAPL` 格式（不应发生但无强制保证），会产生 `ususAAPL` 的错误拼接
- 搜索建议下拉中选中的 `item.symbol` 是带 `us` 前缀的格式，但 `selectSuggestion()` 需要去掉前缀存入 `symbolInput`

### 7. 缓存无过期机制

**严重程度：** 低
**位置：** `frontend/src/App.vue` 中的 `klineCache` 和 `analysisCache`

两个缓存都是 `Map` 对象，在整个页面生命周期内只增不删，没有 TTL（过期时间）和容量上限。

**潜在问题：**

- 长时间使用的用户（如整天开着页面）可能积累大量缓存数据
- 如果用户在盘中查看股票，缓存的 K 线数据不会自动刷新（需要手动重新搜索或刷新页面）
- 切换测试模式/真实模式时不会清除缓存（但会重新请求数据覆盖）

**键格式约束（v0.3.0 更新）：**

- `klineCache` 键：`${symbol}_${period}`（仅股票+周期）
- `analysisCache` 键：`${symbol}_${period}_${startDate}_${endDate}` —— **必须包含起止日期**。区间统计结果依赖日期区间，旧实现键中不含日期会导致"改了日期仍命中旧区间统计、面板不更新"的 bug（已在 v0.3.0 修复）。修改 `handleReCalculate` / 日期选择器逻辑时切勿把日期维度从键中移除。

### 8. 4h K 线的重采样数据量膨胀

**严重程度：** 低
**位置：** `backend/data_provider.py` 第 140 行

```python
fetch_count = count * 4 if need_resample else count
```

为了合成 320 根 4h K 线，需要拉取 1280 根 1h 数据。腾讯的 m60 接口和 Yahoo 的 1h 接口都有数据量限制，可能导致实际可用数据不足。

**Yahoo 1h 数据限制：** Yahoo Finance 的 1h 数据最多保留 730 天（约 2 年），每天约 7 根 1h K 线，共约 5110 根。请求 1280 根在限制范围内，但如果用户需要更多 4h K 线（如调大 count），可能触及上限。

**腾讯 m60 数据限制：** 腾讯的分钟级数据返回量取决于接口参数，实测通常能返回足够数据，但没有官方保证。

### 10. 时间格式混用风险

**严重程度：** 中
**位置：** `frontend/src/App.vue` 第 1302 行（`toChartTime` 函数）

K 线数据的时间字段在不同来源有不同格式：

| 来源 | 日K格式 | 4h格式 |
|------|---------|--------|
| 后端腾讯 | `YYYY-MM-DD` | `YYYY-MM-DD HH:MM` |
| 后端Yahoo | `YYYY-MM-DD` | `YYYY-MM-DD HH:MM` |
| 前端直连腾讯 | `YYYY-MM-DD` | 不支持 |
| mockEngine | `YYYY-MM-DD` | 不支持 |
| generateMockKLines | `YYYY-MM-DD` | 不支持 |

当前 `toChartTime()` 是一个恒等函数（`timeStr => timeStr`），直接透传字符串不做任何转换。这意味着 lightweight-charts 接收到的时间格式完全取决于上游数据源的原始格式。

**已知陷阱：**

- lightweight-charts 首次 `setData` 时会锁定时间格式类型（字符串 vs 时间戳），后续必须保持一致。日K（`YYYY-MM-DD`）和 4h（`YYYY-MM-DD HH:MM`）混用可能引发图表渲染异常
- 历史上 `toChartTime()` 曾是一个复杂的多层解析函数（正则 → Date → 手动拆分），后被简化为恒等函数。如果未来需要重新引入时间格式转换，注意不要破坏格式一致性
- 某些浏览器（特别是旧版 Safari）对 `new Date('YYYY-MM-DD HH:MM')` 的解析行为与 Chrome 不同，可能返回 `NaN`

### 11. 测试覆盖不完整

**严重程度：** 中
**位置：** `backend/test_endpoints.py`

后端测试文件包含 12 个测试用例，覆盖了主要 API 端点和核心算法。但存在以下缺失：

### 14. 重采样频率码必须用起始日锚定 (QS/2QS/YS)

**严重程度：** 高（会导致大周期蜡烛日期错位 + 多出空桶）
**位置：** `backend/data_provider.py` 的 `PERIOD_MAP` 与 `_resample_to_period()`

季K/半年K/年K 由日线 `df.resample(rule)` 聚合。频率码选错会产生严重错位：

| 错误码 | 正确码 | 问题 |
|--------|--------|------|
| `Q`（已废弃→`QE`） | `QS` | 标签为**季末**（如 `2020-03-31`）而非季初，且会冒出一根只含首日的"前导空桶"（如 `2019-12-31`） |
| `2Q`（已废弃→`2QE`） | `2QS` | 同上，半年末标签 + 空桶 |
| `YE`（已废弃，原 `Y`） | `YS` | 年末标签（如 `2020-12-31`）而非年初 `2020-01-01` |

**正确做法：** 一律使用起始日锚定码 `QS` / `2QS` / `YS`，使每根大周期蜡烛的 `time` 标签 = 周期起点（用户期望"每根柱子按年/季/半年来"）。

**注意：** pandas 仍接受废弃码但会打印 `FutureWarning`，且标签语义错误——不要因为"能跑"就沿用。

### 15. Yahoo 沙箱/代理可能退化日线为年聚合桩

**严重程度：** 高（静默产生错标数据）
**位置：** `backend/data_provider.py` 的 `YahooProvider._fetch_yahoo_direct` / `_fetch_yfinance_sync`

在某些网络环境（如沙箱代理）下，Yahoo 的 `interval=1d` 请求会被退化为"每年一根"的聚合桩数据（任意标的都返回约 17 根、相邻 90 天的等值蜡烛，覆盖 1984→2026）。若直接拿它重采样为年K，会丢失全部日线细节并混入错误缩放。

**防线：** `_dispatch_resample` / TencentProvider 重采样入口在聚合前调用 `validate_klines(df, '1d')`（相邻日期中位数间隔 ≤5 天才算日线）。退化数据不通过校验 → 返回空 → Manager 回退到另一数据源（真实日线）。本地 SQLite 缓存读取后也再次校验，污染缓存视为未命中。

**强烈建议：** 在用户本机（有直接外网的真实环境）验证大周期数据，而非仅依赖沙箱代理返回。

### 16. 测试模式前端必须按周期重采样（曾导致"周K=日K"）

**严重程度：** 高（视觉上所有周期都显示成日K）
**位置：** `frontend/src/utils/mockEngine.ts` 的 `generateMockKLines` + `App.vue` 测试模式分支

`generateMockKLines` 只生成**日线**（4500 根）。前端在 v0.3.0 之前**完全没有周期重采样逻辑**——选周K/月K/季K/半年K/年K 时直接把日线交给图表，视觉上"周K=月K=日K"。

**修复：** `mockEngine.ts` 新增 `resampleMockKLines(daily, period)`，按周一归桶(周)/月/季/半年/年聚合（规则与后端 `_resample_to_period` 对齐），`App.vue` 测试模式在生成日线后立即 `klines = resampleMockKLines(klines, currentPeriod.value)`。

**修改陷阱：** 若调整测试模式生成逻辑或新增周期，务必保证 `resampleMockKLines` 覆盖该周期；否则会回归"显示成日K"的 bug。可用 `curl localhost:8000/api/klines?symbol=usAAPL&period=1w&full=true` 对比后端真实粒度（周K 相邻间隔≈7天），确认是前端渲染还是后端数据问题。

### 17. Python 后端改完必须重启进程才生效

**严重程度：** 高（静默不生效，难排查）
**位置：** `backend/main.py` / `data_provider.py` / `analysis.py`

FastAPI 在启动时导入模块（uoricorn worker 加载 `main.py` 及其 import 链），**HMR 只适用于前端 JS，不适用于后端 Python**。改了 `data_provider.py` / `analysis.py` / `main.py` 后若不重启进程，运行中的旧代码仍生效——表现类似"我明明改了代码但 bug 还在"（例如 SNDK 半年K/年K 仍报错，正是因为后端没重启、旧 `≥5` 校验还在跑）。

**正确做法：**
1. `pkill -f main.py`（或停掉旧 worker）
2. 后台重启：`nohup .venv/Scripts/python.exe main.py > backend.log 2>&1 &`
3. 确认生效：`curl -s -m5 http://localhost:8000/api/periods` 返回 200

**前端 JS 不受影响**（Vite dev server HMR 自动热更）。

### 18. 自选栏价格/涨跌幅曾依赖全局 endDate 导致点击跳动（已修复）

**严重程度：** 高（交互层面明显异常）
**位置：** `frontend/src/App.vue`（`updateWatchlistPrices` / `calcLatestPrice`）

**原 bug：** 自选栏每只股票的实时价格和涨跌幅通过 `calcPriceInRange(klines, sd, ed)` 取**全局选中 endDate** 那根 K 线的收盘价与涨跌。点击某只股票时 `selectWatchlist→handleSearch` 会把 `startDate`/`endDate` 重置为该股票记忆的区间，随后 `updateWatchlistPrices()` 用新的 `endDate` 给**所有**自选股重算 → 每点击一次，全部行的数值都跳动（并非真实行情变动）。

**修复（2026-07-15）：**
- 新增 `calcLatestPrice(klines)`：取**日线最后一根**（最新交易日）收盘价 + 与前一日收盘的当日涨跌幅 %，**不再依赖 endDate / 图表周期**。
- 重写 `updateWatchlistPrices()`：一律按**日线(1d)** 拉取并缓存每只股票（演示：`mock_${sym}`；真实：`real_${sym}_1d`），与图表当前周期、选中区间**彻底解耦**。

**修改陷阱（请勿回退）：**
- 自选栏数值必须由各自日线末根唯一决定，**禁止**再引入 `startDate`/`endDate`/`currentPeriod` 参与计算，否则会回归"点击跳动"的 bug。
- 真实模式下 `getKLines(querySym, '1d')` 必须带 `us` 前缀（与 `loadStock` 一致），否则后端 404。
- 演示模式的 `generateMockKLines` 用 `Math.random()`，必须缓存（`watchlistKlineCache`）避免每次调用重新随机生成导致数值跳动。

### 19. 实时行情公开源合规红线（封号 / 403 / 429）

**严重程度：** 高（违反即被数据源封禁 IP，整个实时功能不可用）
**位置：** `backend/quote_config.py`（唯一配置入口）、`backend/quote_provider.py`（`QuoteRateLimiter`）、`frontend/src/utils/quote.ts`（`QuoteEngine`）

**背景：** 实时行情走腾讯免费公开接口 `qt.gtimg.cn/q=`，非商业授权数据源，有严格频控。硬性约束（需求明文规定 + 数据源强制）：

1. **单标的轮询绝对禁止低于 100ms**（`ABSOLUTE_MIN_INTERVAL=0.1s`）。前端若自行写死更短的 `setInterval` 直发，将直接突破红线。
2. **A股本地最低 2s、批量最低 3s**；**港股本地最低 2s**；**美股个股热门服务端 10s 更新**（取 10s 为合规下限）、**盘前/盘后 30–60s**（流动性差，已设 30s 下限）。
3. **美股数据本质是交易所强制 15 分钟延迟快照** —— 刷新再快显示的仍是 15 分钟前的价格，把美股间隔压到 10s 以下既无效又违规。
4. **限流/超时/空数据触发后必须冷却 30s**（`RATE_LIMIT_COOLDOWN=30`），前端同步冷却、不再高频重试；删掉或改短冷却会在被限流后疯狂重试 → 直接封 IP。
5. **单 IP 滑动窗口计数**（配置 `IP_MAX_REQUESTS_PER_WINDOW` + `IP_WINDOW_SECONDS`）超额自动降速（`IP_OVERLOAD_SLOWDOWN_FACTOR`）。本程序为单用户本地应用，正常轮询远低于阈值；若改造为"批量刷几百只/秒"必触发封禁。

**修改陷阱（请勿回退）：**
- 所有间隔/阈值**只**写在 `quote_config.py`，前端 `QuoteEngine` 只从 `/api/quote_config` 读取，**禁止在前端硬编码合规数值**（否则红线约束失效）。
- 后端 `QuoteRateLimiter` 的三重护盾（全局冷却 / 每标的最低间隔 / 单 IP 窗口计数）与 `/api/quote` 的 `_norm_code`（仅小写市场前缀、其余原样）是合规的关键，不要移除或弱化。
- 完整的「可调参数 + 调小下限 + 不可改红线」清单见 `多市场实时行情_改动说明.md` §三 / §四。

### 修改代码时的常见陷阱速查

| 你想做的 | 要注意的 |
|----------|----------|
| 修改支撑阻力算法 | 必须同步修改 `analysis.py` 和 `mockEngine.ts`，否则降级结果与后端不一致 |
| 新增 K 线周期 | 需要修改 6 处：`PERIOD_MAP`（后端）、`SUPPORTED_PERIODS`（后端+前端 api.ts）、`PERIOD_LABELS`（前端 api.ts）、`PERIOD_PARAMS`（后端 analysis.py + 前端 mockEngine.ts）；聚合周期还需设 `resample` 频率码（`QS`/`2QS`/`YS`）并在 `validate_klines` 加粒度阈值 |
| 修改重采样频率码 | 必须用 `QS`/`2QS`/`YS`，严禁 `Q`/`2Q`/`YE`（见第 14 条） |
| 修改股票代码格式 | 检查 App.vue、api.ts、main.py、data_provider.py 中所有格式转换点 |
| 修改 chartUtils.ts 的时间格式 | 检查 `toChartTime()` 是否兼容新格式，特别注意 4h 与日K的区别 |
| 添加新的 API 字段 | 后端 Pydantic 模型和前端 TypeScript 接口需要手动同步 |
| 修改 mockEngine 的接口定义 | 记得同步 api.ts 中的同名接口（或提取到共享 types.ts） |
| 调整测试模式基准价格 | App.vue 的 `basePriceMap` 和 api.ts 的 `volatilityMap` 需要一起改 |
| 修改腾讯 API 的调用方式 | 前端直连和后端使用的是不同端点，不能假设行为一致 |
| 改了测试模式生成/周期逻辑 | 必须同步 `resampleMockKLines`，覆盖所有周期（否则周K/月K 显示成日K，见第 16 条） |
| 改了 data_provider.py / analysis.py / main.py | **必须重启后端进程**（`pkill -f main.py` + 后台 `python main.py`），HMR 只适用于前端（见第 17 条） |
| 改自选栏价格/涨跌幅计算 | 必须基于日线末根（`calcLatestPrice`），禁止引入全局 endDate / currentPeriod，否则点击跳动（见第 18 条） |
