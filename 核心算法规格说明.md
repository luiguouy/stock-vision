# 核心算法规格说明

> 本文档描述 StockVision 两个核心算法的**设计意图、输入输出规格、计算步骤和设计决策**，而非代码复述。目的是让 AI 在修改算法时知道"哪些行为是有意设计的，不能随意改动"。

---

## 1. 自适应支撑阻力位算法 (calculate_sr_levels)

### 1.1 设计意图

从历史 K 线数据中识别出对当前交易最具指导意义的关键支撑位和阻力位。核心思路：价格历史上多次触及的价位会形成"记忆"，这些价位在未来仍具有支撑或阻力作用，但越久远的记忆影响力越弱（时间衰减），越靠近当前价格的位置越有操作价值（临近度加权）。

### 1.2 函数签名

**后端 (Python)：**
```python
def calculate_sr_levels(
    df: pd.DataFrame,        # K线 DataFrame，需含 date/open/high/low/close/volume
    window: int = 5,         # 局部极值窗口（通常由 PERIOD_PARAMS 自动覆盖）
    distance_pct: float = 0.015,  # 已废弃，保留向后兼容
    period: str = '1d'       # K线周期，决定半衰期等参数
) -> List[Dict[str, Any]]
```

**前端 (TypeScript)：**
```typescript
function calculateSRLevels(
    data: KLinePoint[],       // K线数组
    windowSize: number = 5,   // 局部极值窗口
    period: string = '1d'     // K线周期
): SRLevel[]
```

### 1.3 多周期参数表

```
PERIOD_PARAMS (后端 analysis.py / 前端 mockEngine.ts 必须保持同步):

  周期    半衰期(half_life)    极值窗口(window)    含义
  ─────────────────────────────────────────────────────────────
  4h      30 根               5                  约 5 个交易日，捕捉短期波段
  1d      60 根               5                  约 3 个月，日线级别
  1w      26 根               4                  约半年，中期趋势
  1M      12 根               3                  约 1 年，长期结构
  3M      8 根                3                  季K，每根=一个季度
  6M      5 根                3                  半年K，每根=半年
  1Y      4 根                3                  年K，每根=一年
```

**设计决策：** 半衰期以"K 线根数"为单位而非时间，因为不同周期的每根 K 线代表的信息量不同。4h 线 30 根 ≈ 5 天，日线 60 根 ≈ 3 个月，周线 26 根 ≈ 半年，月线 12 根 ≈ 1 年。大周期（季/半年/年）历史根数少，半衰期相应缩短（8/5/4 根）以让"近期"季度仍有足够权重。

**窗口自动覆盖规则：** 当调用方传入 `window=5`（默认值）且 `period != '1d'` 时，自动使用 `PERIOD_PARAMS[period].window`。这意味着外部调用者通常不需要关心窗口参数，只有显式传入非默认值才会覆盖周期默认值。

### 1.4 算法五步流程

#### 第一步：自适应聚类阈值

**目的：** 确定"多近的价格算同一个价位"的合并距离。

```
输入: 最近 20 根 K 线
计算: ATR近似 = mean(high - low) for 最近 20 根
阈值: abs_threshold = max(latest_price × 0.006, min(ATR近似 × 0.8, latest_price × 0.025))
```

**设计决策：**
- 下界 0.6%：防止低价股（如 $5 的股票）合并区间过窄，导致同一价位被拆成多个簇
- 上界 2.5%：防止高价股（如 $500 的股票）合并区间过宽，把不同价位合并到一起
- ATR × 0.8：用近期波动率自适应，波动大时合并区间宽，波动小时合并区间窄
- 使用最近 20 根而非全部数据：只关心"近期"波动率，远古数据对当前交易无指导意义

#### 第二步：局部极值提取

**目的：** 找到 K 线数据中的"波峰"和"波谷"。

```
后端: scipy.signal.argrelextrema(high_values, np.greater_equal, order=window) → 波峰
      scipy.signal.argrelextrema(low_values, np.less_equal, order=window) → 波谷
前端: 滑动窗口遍历，比较 window 范围内的所有 high/low → 波峰/波谷
```

**输出：** 每个极值点包含 `(价格, K线索引)` 两个信息。索引用于后续计算时间衰减。

**设计决策：**
- 波峰用 high，波谷用 low（不用 close）：因为支撑阻力关注的是价格"触及"的极端位置
- `order=window` 表示一个点必须比前后各 window 个点都大/小才算极值，window 越大筛选越严格

#### 第三步：价格合并与时间衰减权重累加

**目的：** 将相近的极值点聚类，并用时间衰减赋予"近期重要、远期次要"的权重。

```
对每个极值点 (price, idx):
  bars_ago = total_len - 1 - idx          # 距离最新 K 线的根数
  weight = 0.5 ^ (bars_ago / half_life)   # 指数衰减，half_life 根前权重 = 0.5

按价格从低到高排序后逐个处理:
  若与已有簇的价格差 < abs_threshold → 合并（加权平均更新价格，累加权重，计数+1）
  否则 → 创建新簇
```

**簇的数据结构：** `{ price: float, weight: float, count: int }`
- `price`：加权平均价格（每次合并都会动态更新）
- `weight`：累计权重（越多次触及 + 越近期 → 权重越大）
- `count`：触及次数

**设计决策：**
- 加权平均而非简单平均：近期触及的价位对"真实价位"的影响更大
- 指数衰减而非线性衰减：符合"记忆遗忘曲线"，近期快速衰减，远期缓慢趋近于零
- 合并时只遍历已有簇（不排序后相邻合并）：实现简单，但可能导致合并顺序依赖——先处理的簇有优先权

#### 第四步：支阻分类与综合打分

**目的：** 根据最新价格判断每个簇是支撑还是阻力，并计算"操作指导分"。

```
对每个簇:
  dist_pct = |cluster_price - latest_price| / latest_price
  proximity_factor = 1 / (1 + 8 × dist_pct)     # 临近度系数 [0, 1]
  score = weight × proximity_factor               # 综合指导分

  分类:
    cluster_price < latest_price → support (支撑)
    cluster_price > latest_price → resistance (阻力)
```

**星级评定：**
```
weight >= 3.5  →  5 星  Strong
weight >= 2.0  →  3 星  Medium
weight <  2.0  →  2 星  Weak
```

**设计决策：**
- 支阻互换机制：同一个价位，在当前价格下方就是支撑，上方就是阻力。不区分"原始支撑"和"原始阻力"——这是有意设计，因为突破后支撑变阻力是常见现象
- 临近度系数公式 `1/(1+8x)`：当距离为 0% 时系数为 1，距离 5% 时系数约 0.44，距离 10% 时约 0.28。系数 8 控制了"远离当前价的价位被惩罚的程度"
- 综合分 = 权重 × 临近度：一个权重很高但距离很远的价位，不如权重中等但紧贴当前价的价位有操作价值

#### 第五步：排序输出

```
支撑位按 score 降序排列，取前 3 个
阻力位按 score 降序排列，取前 3 个
最终输出: [阻力1, 阻力2, 阻力3, 支撑1, 支撑2, 支撑3]
```

**输出规格：**
- 最多 6 个价位（可能更少，如果候选不足）
- 阻力在前，支撑在后
- 每组内部按 score 降序
- 每个元素包含: `price, count, type, stars, level` (不含内部的 `weight, score`)

### 1.5 边界条件与空值处理

| 条件 | 行为 |
|---|---|
| `df` 为空 | 返回 `[]` |
| `len(df) < window * 2 + 1` | 返回 `[]` (数据不足以检测极值) |
| 无极值点 | 返回 `[]` |
| 所有极值价格相同 | 合并为一个簇，根据与最新价关系归为支撑或阻力 |
| 最新价恰好等于某个簇价格 | 该簇既不是支撑也不是阻力，被丢弃 |
| 计算过程异常 | `try/except` 捕获，打印错误，返回 `[]` |

### 1.6 前后端一致性要求

**这是本项目最关键的约束之一：** 前端 `mockEngine.ts` 的 `calculateSRLevels()` 是后端 `analysis.py` 的 1:1 TypeScript 移植，两者必须保持算法完全一致。

修改算法时的检查清单：
1. `PERIOD_PARAMS` 常量是否两端同步？
2. 极值检测逻辑是否一致？(scipy.argrelextrema vs 手动滑动窗口)
3. 合并阈值计算公式是否一致？
4. 时间衰减公式是否一致？
5. 星级阈值 (2.0/3.5) 是否一致？
6. 临近度系数公式 `1/(1+8x)` 是否一致？
7. 输出顺序 (阻力在前) 是否一致？

## 2. 区间最大涨跌幅算法 (calculate_range_stats)

### 2.1 设计意图

在用户指定的日期区间内，找出"最大连贯上涨"和"最大连贯下跌"的幅度、起止日期。用于帮助用户理解某只股票在特定时间段内的极端波动。

### 2.2 函数签名

**后端：**
```python
def calculate_range_stats(
    df: pd.DataFrame,     # K线 DataFrame
    start_date: str,      # 区间开始 'YYYY-MM-DD' 或 'YYYY-MM-DD HH:MM'
    end_date: str         # 区间结束
) -> Dict[str, Any]
```

**前端：**
```typescript
function calculateRangeStats(
    data: KLinePoint[],   // K线数组
    sDate: string,        // 区间开始
    eDate: string         // 区间结束
): RangeStats | null
```

### 2.3 算法流程

#### 最大上涨计算

```
1. 筛选日期区间内的 K 线子集
2. 找到子集中 low 最低的点 (最低谷)
3. 在该最低点之后 (含自身)，找到 high 最高的点 (反弹顶峰)
4. max_rise_pct = (最高点high - 最低点low) / 最低点low × 100
```

**设计决策：** 先找最低点再向后找最高点，确保上涨是"时间上连贯的"——最低点必须发生在最高点之前。

#### 最大下跌计算

```
1. 找到子集中 high 最高的点 (最高峰)
2. 在该最高点之后 (含自身)，找到 low 最低的点 (下跌谷底)
3. max_fall_pct = (最低点low - 最高点high) / 最高点high × 100
   (结果为负数)
```

### 2.4 输出格式

```json
{
  "max_rise": { "pct": 32.61, "start": "2024-01-06", "end": "2024-03-20" },
  "max_fall": { "pct": -14.81, "start": "2024-02-15", "end": "2024-02-28" }
}
```

**注意：** `pct` 上涨为正数，下跌为负数。

### 2.5 边界条件

| 条件 | 后端行为 | 前端行为 |
|---|---|---|
| df/data 为空 | 返回默认值 (pct=0, 空日期) | 返回 `null` |
| start_date/end_date 为空 | 返回默认值 | - |
| 区间内无数据 | 返回默认值 | 返回 `null` |
| 计算异常 | try/except → 返回默认值 | - |

**注意前后端差异：** 后端返回默认值字典 `{pct: 0, start: "", end: ""}`，前端返回 `null`。前端 `api.ts` 在降级调用时做了兜底处理。

### 2.6 前后端一致性要求

与支撑阻力位算法类似，此算法也要求前后端一致，但存在以下差异需要注意：

| 方面 | 后端 | 前端 |
|---|---|---|
| 区间筛选 | 字符串比较 `df['date'] >= start_date` | 字符串比较 `d.time >= sDate` |
| 最低点查找 | `sub_df['low'].idxmin()` (返回 DataFrame 索引) | 手动遍历比较 `.low` |
| 空值返回 | 默认值字典 | `null` |

这些差异不影响计算结果，但 AI 修改时需保持各自的实现风格。

## 3. K线时间聚合重采样算法

季K(3M)/半年K(6M)/年K(1Y) 与 4h 都不是数据源直接返回的粒度，需要由日线（或 1h 线）聚合合成。本节记录两类聚合算法的设计意图与约束。

### 3.1 日历时间聚合 (_resample_to_period，季/半年/年K)

**设计意图：** 季K/半年K/年K 由日线 `pandas.DataFrame.resample()` 按日历时间窗口聚合，每根蜡烛 = 一个季度/半年/年。`OHLCV` 聚合规则：开=窗口首根 open，高=窗口内 max(high)，低=窗口内 min(low)，收=窗口末根 close，量=窗口内 sum(volume)。

**频率码（关键约束）：**

```python
PERIOD_MAP 中的 resample 字段:
  '3M'  → 'QS'     # 季初锚定 (Quarter Start)
  '6M'  → '2QS'    # 半年初锚定 (Semi-Annual Start)
  '1Y'  → 'YS'     # 年初锚定 (Year Start)
```

**严禁使用已废弃码 `Q`/`2Q`/`YE`：** 这些码在 pandas 下会产生"季末/年末标签"且会冒出一个只含首日的"前导空桶"（例如 2020 年的数据却多出一根 `2019-12-31` 错标蜡烛），导致大周期蜡烛日期错位。`QS/2QS/YS` 让每根蜡烛的 `time` 标签对齐周期起点（如 `2020-01-01` 代表 Q1/上半年/全年）。

**前复权顺序（最易踩坑）：**

```
正确顺序:
  1) 对"日线" DataFrame 先做 apply_forward_adjustment() 前复权
  2) 再用 _resample_to_period() 聚合为大周期

错误顺序（会产生拆股缺口）:
  1) 先聚合为大周期
  2) 再前复权 → 由于每根大蜡烛已掩盖了日内的拆股跳变，缺口无法被修正
```

**设计决策：**
- 重采样前必须 `validate_klines(df, '1d')` 确认原始数据确为日线粒度（见「降级策略与容错链路.md」的粒度校验守卫）；若数据源退化（如代理把日线退化为年聚合），直接拒绝重采样并回退到另一数据源
- `DataProviderManager` 对已聚合的 3M/6M/1Y **不再重复前复权**（各 Provider 已在日线阶段完成）
- 这是 `TencentProvider` 的静态方法，`YahooProvider` 的 `_dispatch_resample` 也调用它，避免重复实现

### 3.2 4h K线重采样算法 (_resample_to_4h)

腾讯和 Yahoo 数据源都不直接提供 4 小时 K 线，需要从 1 小时 K 线合成。合成规则：按交易日分组，每 4 根 1h K 线合并为 1 根 4h K 线。

#### 合成规则

```
对于每个交易日:
  按时间排序后，每 4 根 1h K 线为一组:
    open  = 第 1 根的 open
    high  = 4 根中最大的 high
    low   = 4 根中最小的 low
    close = 最后 1 根的 close
    volume = 4 根的 volume 之和
    date  = 第 1 根的 date (保留起始时间)
```

#### 设计决策

- **按交易日分组：** 避免跨交易日（隔夜缺口）合并产生失真。美股交易时间约 6.5 小时，每天约 6-7 根 1h K 线，按 4 根分组后每天产生 1-2 根 4h K 线
- **复用机制：** YahooProvider 直接调用 `TencentProvider._resample_to_4h()` 静态方法，避免重复实现
- **这是静态方法：** 不需要实例化 TencentProvider 即可调用

#### 数据量放大

由于需要从 1h 数据合成 4h，拉取数据时需要 4 倍数量：
```python
fetch_count = count * 4 if need_resample else count
# 例如需要 320 根 4h K 线 → 实际拉取 1280 根 1h K 线
```

## 4. 随机游走 Mock 数据生成 (generateMockKLines)

### 4.1 设计意图

在后端完全不可用时提供测试数据，让前端 UI 可以独立开发和调试。

### 4.2 生成规则

```
初始价格 = basePrice (按股票不同有预设值)
每日:
  change = (random - 0.5) × 2 × volatility    # [-volatility, +volatility]
  open = 前一日 close
  close = open × (1 + change)
  high = max(open, close) × (1 + random × volatility × 0.5)
  low  = min(open, close) × (1 - random × volatility × 0.5)
  volume = random(5M, 55M)
```

### 4.3 各股票波动率参数

```
AAPL: 2.0%   NVDA: 3.5%   TSLA: 4.0%   MSFT: 1.8%
GOOGL: 2.2%  AMZN: 2.5%   META: 3.0%   AMD:  3.8%
MU:   4.5%   SNDK: 3.2%   其他: 2.5%
```

### 4.4 约束

- 跳过周末 (Saturday/Sunday)
- 生成 320 个交易日的数据
- 这是纯前端函数，不涉及任何网络请求
